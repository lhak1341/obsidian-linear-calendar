# No shared CalendarWidget factory between mountMonthStrip and LinearCalendarView

`mountMonthStrip` (public API) and `LinearCalendarView` both call `new CalendarRenderer(...)`, which looks like shared setup logic worth extracting. We decided not to introduce a `createCalendarWidget()` factory because the two callers need meaningfully different resize policies: `mountMonthStrip` guards on actual width change (`w !== lastWidth && w > 0`) to avoid false triggers in the dashboard container; `LinearCalendarView` uses `debounce(..., 200)` to rate-limit continuous pane-resize events. A factory that encapsulates both would need to parameterize resize behavior, making it more complex than just calling `new CalendarRenderer()` directly. The deletion test confirms it: remove the factory and both callers simply construct `CalendarRenderer` themselves — no complexity concentrates behind the seam. The `CalendarRenderer` constructor is the right level of abstraction here.
